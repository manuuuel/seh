# GitHub Repo Hardening Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Harden the now-public `manuuuel/seh` repo per
`docs/superpowers/specs/2026-07-10-github-hardening-design.md`: branch
protection on `main`, `CONTRIBUTING.md`, PR/issue templates, and removal of
the now-superseded `## Contributing` README section.

**Architecture:** Four independent tasks — three are new/modified files
(`CONTRIBUTING.md`, `.github/PULL_REQUEST_TEMPLATE.md`,
`.github/ISSUE_TEMPLATE/*.md`), reordered/edited into a single PR; the
fourth configures GitHub branch protection via the REST API (no repo file
changes, but must be done by someone with admin access — the session
controller, not a subagent, since it requires the authenticated `gh` CLI
session and is an irreversible-by-force-push-prevention repo setting change).

**Tech Stack:** Markdown, GitHub Actions config (existing, unmodified),
GitHub REST API via `gh api`.

## Global Constraints

- `CONTRIBUTING.md` must not describe the brainstorming/spec/plan authoring
  pipeline — it points at `AGENTS.md` as the single binding contract.
- Branch protection: 1 required approving review, required status checks
  (`test (18)`, `test (20)`, `test (22)`), `enforce_admins: false` (admin
  bypass allowed), force-push and branch deletion blocked, `strict: false`
  (not requiring up-to-date branches).
- No `CODE_OF_CONDUCT.md`, `SECURITY.md`, or `.github/dependabot.yml` in
  this plan (out of scope per spec).
- README changes in this plan are limited to removing the `## Contributing`
  heading and its Table of Contents entry — nothing else in README changes.

- [ ] **Step 1: Write `CONTRIBUTING.md`**

```markdown
# Contributing to seh

Thanks for your interest in contributing.

## Before you contribute

This project is governed by its own `AGENTS.md`. Read it before
contributing — it defines the workflow (plan before code, no
implementation until reviewed), commit conventions, branching, and quality
gates that apply to every PR.

`AGENTS.md` is a gitignored symlink to `.seh/AGENTS.md`, generated by
`seh sync`. If you're changing project guidelines, edit the sources under
`.seh/` — never `AGENTS.md` directly — then run `seh sync` to regenerate.

## Local development

```bash
npm install
npm run build    # compile to dist/
npm test         # run the test suite
npm run dev      # run the CLI via tsx without building
npm run try      # sandboxed demo against a throwaway HOME
```

## Branching & commits

- Branch name reflects intent: `feat/…`, `fix/…`, `docs/…`, `chore/…`.
- Commit messages follow [Conventional Commits](https://www.conventionalcommits.org/):
  `type(scope): subject`, ≤ 50 chars, imperative mood.
  Types: `feat | fix | refactor | test | docs | chore | ci`.

## Pull requests

- One focused change per PR.
- `npm run build` and `npm test` must pass locally before you open a PR.
- CI must pass and at least one approving review is required before merge
  (enforced by branch protection on `main`).
- Use the PR template — it's applied automatically when you open a PR.

## Reporting issues

Use the issue templates (bug report or feature request) — they're offered
automatically when you open a new issue.
```

### Task 4: Configure branch protection on `main`

**Files:** None (GitHub repository setting, not a file in this repo).

**Interfaces:** N/A.

This task is executed by the session controller directly (not dispatched to
a subagent), since it requires the authenticated `gh` CLI session tied to
repo admin access, and is a live repository setting rather than a
reviewable file diff.

- [ ] **Step 1: Determine the exact required status check names**

Run: `gh pr checks 3 2>&1 || gh run list --workflow=ci.yml --limit 1 --json name,displayTitle`

The CI workflow (`.github/workflows/ci.yml`) runs a matrix job named `test`
over `node-version: [18, 20, 22]`; GitHub reports each matrix leg as a
separate check named `test (18)`, `test (20)`, `test (22)` (already
observed in this repo's PR #3 checks). Use these three exact strings.

- [ ] **Step 2: Apply branch protection**

```bash
gh api repos/manuuuel/seh/branches/main/protection \
  --method PUT \
  -H "Accept: application/vnd.github+json" \
  -f 'required_status_checks[strict]=false' \
  -f 'required_status_checks[contexts][]=test (18)' \
  -f 'required_status_checks[contexts][]=test (20)' \
  -f 'required_status_checks[contexts][]=test (22)' \
  -F 'enforce_admins=false' \
  -f 'required_pull_request_reviews[required_approving_review_count]=1' \
  -F 'required_pull_request_reviews[dismiss_stale_reviews]=false' \
  -F 'required_pull_request_reviews[require_code_owner_reviews]=false' \
  -F 'restrictions=null' \
  -F 'allow_force_pushes=false' \
  -F 'allow_deletions=false'
```

- [ ] **Step 3: Verify**

Run: `gh api repos/manuuuel/seh/branches/main/protection`

Expected JSON includes:
- `"required_status_checks":{"strict":false,"contexts":["test (18)","test (20)","test (22)"]...}`
- `"enforce_admins":{"enabled":false}`
- `"required_pull_request_reviews":{"required_approving_review_count":1,...}`
- `"allow_force_pushes":{"enabled":false}`
- `"allow_deletions":{"enabled":false}`

- [ ] **Step 4: No commit** (this task changes no files; nothing to commit)
