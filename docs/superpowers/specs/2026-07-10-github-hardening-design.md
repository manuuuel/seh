# GitHub repo hardening — design

## Context

Sub-project 2 of a 4-part public-readiness effort (README polish → **GitHub
hardening/CONTRIBUTING** → stack content enrichment → history rewrite/memory
migration). The repo (`manuuuel/seh`) was private and has just been made
public. This spec covers branch protection and contribution-hygiene files
only.

## Goals

- Protect `main` from unreviewed/unchecked direct changes now that the repo
  is public, while keeping the solo-maintainer workflow unblocked.
- Add `CONTRIBUTING.md` as a root-level community file (same tier as
  `LICENSE`), auto-detected by GitHub — not referenced from the README body
  or its Table of Contents.
- Add a PR template and issue templates so incoming contributions arrive
  with the information needed to review/triage them.
- Remove the `## Contributing` section and its TOC entry added in
  sub-project 1 (superseded by GitHub's native community-file detection).

## Non-goals

- No `CODE_OF_CONDUCT.md`, `SECURITY.md`, or `.github/dependabot.yml`
  (deferred — user chose the "Standard" tier, not "Full").
- No stack module content changes (sub-project 3).
- No git history changes (sub-project 4).
- `CONTRIBUTING.md` does not describe the brainstorming/spec/plan authoring
  pipeline — that workflow is being migrated into `.seh/memory` separately
  and is not a repo-level contribution rule. `CONTRIBUTING.md` instead
  points at `AGENTS.md` as the single binding contract for contributors.

## Branch protection on `main`

Configured via GitHub branch protection (REST API /
`gh api repos/manuuuel/seh/branches/main/protection`, since the repo is now
public and the endpoint is available):

- `required_pull_request_reviews.required_approving_review_count: 1`
- `required_status_checks`: the `test` job from `.github/workflows/ci.yml`
  (all matrix legs: Node 18, 20, 22) must pass; `strict: false` (not
  requiring branches to be up to date with `main` before merge — user
  explicitly excluded this requirement)
- `enforce_admins: false` — admins (the repo owner) can bypass the PR/review
  requirement to self-merge when there is no second reviewer available; the
  rule is still fully enforced for any non-admin contributor
- `allow_force_pushes: false`
- `allow_deletions: false`
- No restrictions on who can push (not needed while `enforce_admins: false`
  already gates non-admins through the PR requirement)

## `CONTRIBUTING.md`

Root file, plain markdown, sections:

1. **Before you contribute** — "This project is governed by its own
   `AGENTS.md`. Read it before contributing — it defines the workflow (plan
   before code, no implementation until reviewed), commit conventions,
   branching, and quality gates that apply to every PR."
2. **Dogfooding note** — `AGENTS.md` is a gitignored symlink to
   `.seh/AGENTS.md`, generated by `seh sync`; contributors editing
   guidelines edit `.seh/*.md` — never `AGENTS.md` directly — and re-run
   `seh sync` (confirmed: `AGENTS.md` is untracked in this repo, listed in
   `.gitignore` under "seh — generated tool symlinks").
3. **Local development** — `npm install`, `npm run build`, `npm test`,
   `npm run dev`, `npm run try` (sandboxed demo), lifted verbatim from the
   README's existing Development section (no duplication of prose — a
   short pointer plus the exact commands).
4. **Branching & commits** — branch naming (`feat/`, `fix/`, `docs/`,
   `chore/`), Conventional Commits format (`type(scope): subject`),
   mirroring the global AGENTS.md rules already in force for this repo.
5. **Pull requests** — one focused change per PR; `npm run build` and
   `npm test` must pass locally before opening; CI must pass before merge
   (enforced by branch protection); at least one approving review required
   (enforced by branch protection, admin-bypassable).

## `.github/PULL_REQUEST_TEMPLATE.md`

Sections: **Summary** (what/why), **Changes** (bullet list), **How
verified** (commands run + output/pass status), **Related issue** (optional,
`Closes #N` or "N/A").

## `.github/ISSUE_TEMPLATE/`

Two templates using GitHub's issue-form or markdown format (markdown,
matching this repo's existing plain style):

- `bug_report.md` — Description, Steps to reproduce, Expected behavior,
  Actual behavior, Environment (`seh --version`, OS, Node version).
- `feature_request.md` — Problem statement, Proposed solution, Alternatives
  considered.

## README changes

Remove (added in sub-project 1, now superseded):
- The `## Contributing` section (one line: "See CONTRIBUTING.md…").
- Its Table of Contents entry.

No other README changes in this sub-project.

## Testing / verification

- No automated tests apply to markdown/config-only changes. Verification is
  manual:
  - `gh api repos/manuuuel/seh/branches/main/protection` reflects the
    settings above after configuration.
  - A test PR from a branch confirms: direct push to `main` is rejected,
    CI check is required, merge is blocked pending 1 approval unless merged
    via admin bypass.
  - `npm run build && npm test` still pass (no code touched).
  - README TOC no longer references Contributing; heading is gone.

## Risks

- Low risk: config + markdown only, fully reversible (branch protection can
  be edited or removed; files can be deleted).
- `enforce_admins: false` means the protection is not airtight against the
  repo owner's own mistakes (e.g., accidentally using an admin-bypass merge
  on a PR that should have been reviewed). Accepted tradeoff for a
  solo-maintainer project per explicit user choice.
